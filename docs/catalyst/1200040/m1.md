# Catalyst 1200040 milestone 1

Proposal: [Blaze: Fully Featured Ledger Emulator, Transaction Builder for the Chang Hard Fork and PlutusV3](https://milestones.projectcatalyst.io/projects/1200040)

Current Statement of Milestones: `soms.id=10022`, Milestone 1, "Research and Specification Development".

## What M1 claims

The current SoM requires:

- Public detailed issue tracking and iterative planning for all required emulator features for Chang hard fork and PlutusV3 support.
- Technical analysis of current implementations via `cardano-node` and transaction builders such as CML and Lucid.
- Performance benchmarks and testing criteria.
- A proposed feature set covering Chang hard fork and PlutusV3 capabilities.
- Public technical-analysis discussion, viewable performance results, and peer review from at least two senior Cardano developers.

## Planning

M1 is a research and specification milestone. The planned emulator work is divided into these public planning tracks:

- Conway transaction-body support for voting procedures, proposal procedures, treasury value, and donation fields.
- PlutusV3 script serialization and language-use tracking in transaction witnesses.
- DRep registration, update, unregistration, vote delegation, and script-credential witness handling.
- Governance proposal admission for parameter changes, treasury withdrawals, committee updates, new constitution, no confidence, hard fork initiation, and info actions.
- Local epoch-state modelling for DRep state, committee state, constitution, proposal records, snapshots, expiry, deposits, treasury, ratification, and enactment.
- Developer-facing test inspection for proposal status, DReps, committee state, vote tallies, treasury, and snapshots.
- Benchmark scenarios and regression criteria that compare governance flows with ordinary emulator operations.

The M1 evidence is limited to showing that these tracks are planned, analysed, and reviewable. It does not treat every feature in the list as already delivered.

| `cardano-cli`                      | `f017b7151459474675d0939199edafc5cc3b9dcf` | Conway governance action, vote, DRep, proposal-script, and vote-script command implementations      |

The reviewed libraries cover a lot of Conway transaction construction and witness handling. None of them provides the full local governance-emulator behavior described by this proposal: proposal admission, vote validation, DRep and committee state, snapshots, tallies, ratification, enactment, expiry, refunds, treasury effects, and inspectable RPC state.

### Ledger and CLI requirements

The Conway transaction format makes governance part of the transaction body. CML's Conway CDDL defines `voting_procedures` as voter-to-action voting maps and `proposal_procedure` as deposit, return reward account, governance action, and anchor ([governance.cddl](https://github.com/dcSpark/cardano-multiplatform-lib/blob/a34592e1e0d3c1e958d72f05c72282d788403aa5/specs/conway/governance.cddl#L2-L14)). The same CDDL lists the seven action families covered by the proposal: parameter change, hard fork initiation, treasury withdrawals, no confidence, committee update, new constitution, and info action ([governance.cddl](https://github.com/dcSpark/cardano-multiplatform-lib/blob/a34592e1e0d3c1e958d72f05c72282d788403aa5/specs/conway/governance.cddl#L16-L25)). Voters are not a single credential type; the CDDL distinguishes committee hot key/script, DRep key/script, and stake-pool key voters ([governance.cddl](https://github.com/dcSpark/cardano-multiplatform-lib/blob/a34592e1e0d3c1e958d72f05c72282d788403aa5/specs/conway/governance.cddl#L70-L95)).

The certificate surface expands accordingly. Conway certificates include vote delegation, stake-and-vote delegation, vote registration/delegation, committee hot authorization/resignation, DRep registration, DRep unregistration, and DRep update ([certs.cddl](https://github.com/dcSpark/cardano-multiplatform-lib/blob/a34592e1e0d3c1e958d72f05c72282d788403aa5/specs/conway/certs.cddl#L14-L28)). The DRep certificate tags carry explicit deposits and anchors where appropriate ([certs.cddl](https://github.com/dcSpark/cardano-multiplatform-lib/blob/a34592e1e0d3c1e958d72f05c72282d788403aa5/specs/conway/certs.cddl#L76-L81)), and `d_rep` values may be key, script, `always_abstain`, or `always_no_confidence` ([certs.cddl](https://github.com/dcSpark/cardano-multiplatform-lib/blob/a34592e1e0d3c1e958d72f05c72282d788403aa5/specs/conway/certs.cddl#L83-L91)).

The Conway transaction body adds `voting_procedures`, `proposal_procedures`, `current_treasury_value`, and `donation` fields ([transaction.cddl](https://github.com/dcSpark/cardano-multiplatform-lib/blob/a34592e1e0d3c1e958d72f05c72282d788403aa5/specs/conway/transaction.cddl#L27-L30)). It also adds PlutusV3 language-view handling and requires the corresponding cost model when a Plutus language is used ([transaction.cddl](https://github.com/dcSpark/cardano-multiplatform-lib/blob/a34592e1e0d3c1e958d72f05c72282d788403aa5/specs/conway/transaction.cddl#L104-L109)).

`cardano-cli` shows how users construct these objects. Its Conway governance action runner dispatches separate commands for constitution, protocol-parameter update, treasury withdrawal, committee update, no-confidence, hard-fork, info, and view actions ([Actions/Run.hs](https://github.com/IntersectMBO/cardano-cli/blob/f017b7151459474675d0939199edafc5cc3b9dcf/cardano-cli/src/Cardano/CLI/EraBased/Governance/Actions/Run.hs#L41-L60)). The implementations create proposal procedures from deposits, return stake credentials, anchors, optional previous governance action IDs, and action-specific fields for no-confidence/new constitution ([Actions/Run.hs](https://github.com/IntersectMBO/cardano-cli/blob/f017b7151459474675d0939199edafc5cc3b9dcf/cardano-cli/src/Cardano/CLI/EraBased/Governance/Actions/Run.hs#L119-L223)), committee updates and protocol-parameter updates ([Actions/Run.hs](https://github.com/IntersectMBO/cardano-cli/blob/f017b7151459474675d0939199edafc5cc3b9dcf/cardano-cli/src/Cardano/CLI/EraBased/Governance/Actions/Run.hs#L226-L342)), and treasury withdrawal plus hard-fork initiation ([Actions/Run.hs](https://github.com/IntersectMBO/cardano-cli/blob/f017b7151459474675d0939199edafc5cc3b9dcf/cardano-cli/src/Cardano/CLI/EraBased/Governance/Actions/Run.hs#L386-L480)).

The CLI also treats governance scripts as first-class script purposes. Vote files can carry key witnesses, simple scripts, Plutus scripts, reference scripts, redeemers, and execution units; the code notes that script-witnessed vote files are constrained to avoid ambiguous redeemer-map construction ([Vote/Read.hs](https://github.com/IntersectMBO/cardano-cli/blob/f017b7151459474675d0939199edafc5cc3b9dcf/cardano-cli/src/Cardano/CLI/EraBased/Script/Vote/Read.hs#L27-L105)). Proposal files have equivalent simple/reference/Plutus witness support and wrap Plutus witnesses as proposing-script witnesses ([Proposal/Read.hs](https://github.com/IntersectMBO/cardano-cli/blob/f017b7151459474675d0939199edafc5cc3b9dcf/cardano-cli/src/Cardano/CLI/EraBased/Script/Proposal/Read.hs#L27-L118)).

DRep registration requires CLI key generation, registration certificate creation, transaction build/sign/submit, and the current DRep deposit queried from governance state ([DRep.hs](https://github.com/IntersectMBO/cardano-node/blob/a53156c39ae56b3b24bab9433def3143cea2b216/cardano-testnet/src/Testnet/Process/Cli/DRep.hs#L63-L103), [DRep.hs](https://github.com/IntersectMBO/cardano-node/blob/a53156c39ae56b3b24bab9433def3143cea2b216/cardano-testnet/src/Testnet/Process/Cli/DRep.hs#L231-L248)). Voting requires separate vote files, a transaction body with `--vote-file`, witness overrides, signing, submission, and later lookup of the governance action ID/index in epoch state ([DRep.hs](https://github.com/IntersectMBO/cardano-node/blob/a53156c39ae56b3b24bab9433def3143cea2b216/cardano-testnet/src/Testnet/Process/Cli/DRep.hs#L147-L209), [DRep.hs](https://github.com/IntersectMBO/cardano-node/blob/a53156c39ae56b3b24bab9433def3143cea2b216/cardano-testnet/src/Testnet/Process/Cli/DRep.hs#L341-L425)). The testnet helper queries live `NewEpochState` for DRep state, governance state, DRep deposit, governance action deposit, treasury, protocol params, and governance action lifetime ([Query.hs](https://github.com/IntersectMBO/cardano-node/blob/a53156c39ae56b3b24bab9433def3143cea2b216/cardano-testnet/src/Testnet/Components/Query.hs#L515-L596), [Query.hs](https://github.com/IntersectMBO/cardano-node/blob/a53156c39ae56b3b24bab9433def3143cea2b216/cardano-testnet/src/Testnet/Components/Query.hs#L650-L672)).

### CML analysis

CML provides Conway transaction and protocol-parameter structures. Its multi-era Rust helpers expose Conway-only `voting_procedures` and `proposal_procedures` accessors ([utils.rs](https://github.com/dcSpark/cardano-multiplatform-lib/blob/a34592e1e0d3c1e958d72f05c72282d788403aa5/multi-era/rust/src/utils.rs#L746-L766)). It exposes governance action validity period, governance action deposit, DRep deposit, and DRep inactivity period as Conway protocol-parameter update fields ([utils.rs](https://github.com/dcSpark/cardano-multiplatform-lib/blob/a34592e1e0d3c1e958d72f05c72282d788403aa5/multi-era/rust/src/utils.rs#L1179-L1212)). The WASM helpers expose Conway certificate downcasts for vote delegation and DRep certificates ([utils.rs](https://github.com/dcSpark/cardano-multiplatform-lib/blob/a34592e1e0d3c1e958d72f05c72282d788403aa5/multi-era/wasm/src/utils.rs#L307-L383)), plus transaction accessors for voting/proposal procedures ([utils.rs](https://github.com/dcSpark/cardano-multiplatform-lib/blob/a34592e1e0d3c1e958d72f05c72282d788403aa5/multi-era/wasm/src/utils.rs#L649-L654)).

CML does not provide a local ledger-state emulator. It shows what a Conway transaction contains and how those fields are serialized or inspected. It does not maintain DRep registration state, committee hot credentials, stake snapshots, proposal expiry, ratification thresholds, enactment queues, deposit refunds, or treasury withdrawals.

### Mesh analysis

Mesh has substantial builder support around Conway governance. Its V2 builder has explicit PlutusV3 spend/mint/withdraw methods and `votePlutusV1`, `votePlutusV2`, and `votePlutusV3` helpers ([mesh-tx-builder-v2/index.ts](https://github.com/MeshJS/mesh/blob/3df879e48f400f19eea0ebddfb00099b0809f484/packages/mesh-transaction/src/mesh-tx-builder-v2/index.ts#L239-L335)). The core builder supports voting scripts, reference scripts, vote redeemers, proposal creation, proposal scripts, proposal reference scripts, and proposal redeemers ([tx-builder-core.ts](https://github.com/MeshJS/mesh/blob/3df879e48f400f19eea0ebddfb00099b0809f484/packages/mesh-transaction/src/mesh-tx-builder/tx-builder-core.ts#L941-L1235)). It also has DRep registration, deregistration, and update certificate builder methods ([tx-builder-core.ts](https://github.com/MeshJS/mesh/blob/3df879e48f400f19eea0ebddfb00099b0809f484/packages/mesh-transaction/src/mesh-tx-builder/tx-builder-core.ts#L1322-L1378)).

Mesh also implements several transaction-construction details tied to the proposal surface: it sorts voters by committee/DRep/SPO order and then by credential details while leaving proposals in insertion order ([mesh-tx-builder/index.ts](https://github.com/MeshJS/mesh/blob/3df879e48f400f19eea0ebddfb00099b0809f484/packages/mesh-transaction/src/mesh-tx-builder/index.ts#L447-L523)), produces `VOTE` redeemer tags with indexes for script votes ([mesh-tx-builder/index.ts](https://github.com/MeshJS/mesh/blob/3df879e48f400f19eea0ebddfb00099b0809f484/packages/mesh-transaction/src/mesh-tx-builder/index.ts#L592-L615)), derives required witnesses for DRep certificates and basic votes ([mesh-tx-builder/index.ts](https://github.com/MeshJS/mesh/blob/3df879e48f400f19eea0ebddfb00099b0809f484/packages/mesh-transaction/src/mesh-tx-builder/index.ts#L1158-L1176), [mesh-tx-builder/index.ts](https://github.com/MeshJS/mesh/blob/3df879e48f400f19eea0ebddfb00099b0809f484/packages/mesh-transaction/src/mesh-tx-builder/index.ts#L1238-L1262)), and adds DRep/proposal deposits into transaction deposit accounting ([mesh-tx-builder/index.ts](https://github.com/MeshJS/mesh/blob/3df879e48f400f19eea0ebddfb00099b0809f484/packages/mesh-transaction/src/mesh-tx-builder/index.ts#L1294-L1312)). Its type layer models governance proposal status fields, protocol-parameter governance thresholds, deposits, action validity period, and DRep inactivity ([governance.ts](https://github.com/MeshJS/mesh/blob/3df879e48f400f19eea0ebddfb00099b0809f484/packages/mesh-common/src/types/governance.ts#L5-L150)).

Mesh remains a transaction builder and chain-client toolkit. The reviewed source covers governance transaction construction, witness derivation, redeemer indexing, and deposit accounting. It does not show deterministic local ratification, enactment, expiry, refund, treasury, or governance-query simulation.

### Lucid Evolution analysis

Lucid Evolution covers much of the high-level DRep/delegation builder surface. Its `TxBuilder` interface includes DRep register/deregister/update, vote delegation to DReps, register-and-delegate helpers, treasury donation, committee hot authorization/resignation, and `attach.VoteValidator` / `attach.ProposeValidator` ([TxBuilder.ts](https://github.com/Anastasia-Labs/lucid-evolution/blob/2c9056958f8dc4e78a3207e0246f8a31bc014c2f/packages/lucid/src/tx-builder/TxBuilder.ts#L580-L686)). The implementation records DRep registration and deregistration actions ([TxBuilder.ts](https://github.com/Anastasia-Labs/lucid-evolution/blob/2c9056958f8dc4e78a3207e0246f8a31bc014c2f/packages/lucid/src/tx-builder/TxBuilder.ts#L937-L1008)), treasury donation ([TxBuilder.ts](https://github.com/Anastasia-Labs/lucid-evolution/blob/2c9056958f8dc4e78a3207e0246f8a31bc014c2f/packages/lucid/src/tx-builder/TxBuilder.ts#L1099-L1124)), vote delegation and register-and-delegate flows ([TxBuilder.ts](https://github.com/Anastasia-Labs/lucid-evolution/blob/2c9056958f8dc4e78a3207e0246f8a31bc014c2f/packages/lucid/src/tx-builder/TxBuilder.ts#L1186-L1303)). Its core types include DRep and governance deposits in protocol parameters, optional treasury provider access, `vote` and `propose` redeemer tags, PlutusV3, and vote/propose validators ([types.ts](https://github.com/Anastasia-Labs/lucid-evolution/blob/2c9056958f8dc4e78a3207e0246f8a31bc014c2f/packages/core-types/src/types.ts#L5-L119), [types.ts](https://github.com/Anastasia-Labs/lucid-evolution/blob/2c9056958f8dc4e78a3207e0246f8a31bc014c2f/packages/core-types/src/types.ts#L185-L199), [types.ts](https://github.com/Anastasia-Labs/lucid-evolution/blob/2c9056958f8dc4e78a3207e0246f8a31bc014c2f/packages/core-types/src/types.ts#L391-L399)).

Lucid also translates Conway DRep/delegation certificates into Plutus script context data ([ScriptContext.ts](https://github.com/Anastasia-Labs/lucid-evolution/blob/2c9056958f8dc4e78a3207e0246f8a31bc014c2f/packages/lucid/src/script-context/ScriptContext.ts#L266-L420)) and extracts required witness key hashes from vote delegation and DRep certificates ([wallet.ts](https://github.com/Anastasia-Labs/lucid-evolution/blob/2c9056958f8dc4e78a3207e0246f8a31bc014c2f/packages/wallet/src/wallet.ts#L188-L285)). Its public governance docs focus on vote delegation, predefined DRep strategies, DRep registration/deregistration/update, and script DReps ([governance.mdx](https://github.com/Anastasia-Labs/lucid-evolution/blob/2c9056958f8dc4e78a3207e0246f8a31bc014c2f/docs/pages/documentation/deep-dives/governance.mdx#L22-L202)).

Lucid's emulator docs state that the emulator is simplified and does not implement all Cardano features, with complex governance operations named as an example ([emulator.mdx](https://github.com/Anastasia-Labs/lucid-evolution/blob/2c9056958f8dc4e78a3207e0246f8a31bc014c2f/docs/pages/documentation/deep-dives/emulator.mdx#L177-L191)).

### Findings from the review


| Upstream behavior                                                                                                                              | M1 planning implication                                                                                                               |
| ---------------------------------------------------------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------- |
| Conway transaction bodies contain voting procedures and proposal procedures                                                                    | The emulator feature plan includes governance transaction-body fields instead of treating governance as off-chain metadata.           |
| PlutusV3 is serializable and script language use is tracked in Conway transactions                                                             | The feature plan includes PlutusV3 script witnesses and cost-model handling.                                                          |
| DRep certificates and vote delegation are Conway certificates with key or script credentials                                                   | The feature plan includes DRep lifecycle, vote delegation, and script-credential witness paths.                                       |
| Vote and proposal scripts use governance redeemer purposes                                                                                     | The feature plan separates `Voting` and `Proposing` redeemers from spend, mint, certificate, and reward redeemers.                    |
| Proposals carry deposit, return account, action kind, anchor, optional policy hash, and sometimes previous action linkage                      | The feature plan includes proposal admission checks, bootstrap handling, policy hashes, and previous-action references.               |
| Governance state changes across epochs                                                                                                         | The feature plan includes snapshots, expiry, ratification, enactment, deposits, refunds, treasury state, and last enacted action IDs. |
| `cardano-testnet` relies on live node state queries for DRep state, governance state, deposits, treasury, protocol params, and action lifetime | The feature plan includes local inspection points for the same classes of state so SDK tests do not need a live node.                 |
| M1 asks for benchmark and test criteria                                                                                                        | The plan includes baseline emulator scenarios, governance-flow scenarios, coverage thresholds, and regression checks.                 |


## Proposed feature set

This is the proposed feature set produced by M1 planning. Implementation can be split across later milestones.

- Governance action submission for parameter changes, treasury withdrawals, committee updates, new constitution, no confidence, hard fork initiation, and info actions.
- DRep registration, re-registration, deregistration/expiry handling, vote delegation, key DRep voting, and script DRep voting.
- Committee membership, quorum, and hot credential handling.
- Stake snapshot construction, DRep/SPO/committee tallying, threshold checks, ratification, enactment, delayed enactment for no-confidence, expiry, and deposit refunds.
- Proposal-policy script witness validation for parameter-change and treasury-withdrawal actions.
- PlutusV3-capable script handling through core exports and transaction witness serialization.
- RPC endpoints for governance state inspection and test orchestration.
- Benchmarks and executable docs for SDK test-harness use.

Out of proposed emulator scope:

- Network synchronization, mempool behavior, or chain selection.
- Live-node governance submission automation.
- Full formal proof against the Conway ledger spec.

This scope is narrower than `cardano-node`, but deeper than a transaction-builder comparison. It targets the stateful governance behavior that dApp and SDK developers test locally.

## Performance baseline and test criteria

M1 uses a small benchmark matrix to set the baseline for later emulator delivery:

- register mocked wallet
- simple payment
- script lock and spend
- parameter-change governance flow with DRep registration, proposal submission, voting, and epoch advancement

The benchmark source is [packages/blaze-emulator/benchmarks/index.ts](../../../packages/blaze-emulator/benchmarks/index.ts#L48-L220). Fresh local baseline on 2026-06-04:

```sh
pnpm --filter @blaze-cardano/emulator bench
```


| Task                             | Latency avg   | Throughput avg |
| -------------------------------- | ------------- | -------------- |
| register mocked wallet           | 258,294 ns    | 3,971 ops/s    |
| simple payment                   | 3,192,224 ns  | 352 ops/s      |
| script lock -> spend             | 10,696,025 ns | 97 ops/s       |
| parameter change governance flow | 12,849,683 ns | 82 ops/s       |


Testing criteria for later delivery:

- Unit tests cover each governance action family in the proposed feature set.
- Tests cover DRep lifecycle, vote delegation, key and script DRep voting, proposal-policy scripts, and PlutusV3 witness serialization.
- Epoch-transition tests cover snapshots, expiry, refunds, ratification, enactment, no-confidence delay, and treasury effects.
- RPC or equivalent inspection tests cover proposal status, DRep state, committee state, tallies, treasury, and snapshots.
- Emulator package coverage stays above the 80% milestone target for new feature work.
- Benchmarks report the same baseline scenarios so governance overhead can be compared with ordinary emulator operations.

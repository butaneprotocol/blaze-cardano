# Catalyst 1200040 Milestone 1 Evidence

Proposal: [Blaze: Fully Featured Ledger Emulator, Transaction Builder for the Chang Hard Fork and PlutusV3](https://milestones.projectcatalyst.io/projects/1200040)

Current Statement of Milestones: `soms.id=10022`, Milestone 1, "Research and Specification Development".

## M1 Claims

The current SoM requires:

- Public detailed issue tracking and iterative planning for all required emulator features for Chang hard fork and PlutusV3 support.
- Technical analysis of current implementations via `cardano-node` and transaction builders such as CML and Lucid.
- Performance benchmarks and testing criteria.
- A proposed feature set covering Chang hard fork and PlutusV3 capabilities.
- Public technical-analysis discussion, viewable performance results, and peer review from at least two senior Cardano developers.

## Feature Scope Covered

Blaze M1 scope is implemented around Conway-era governance and PlutusV3-capable transaction emulation:

- Core governance types and PlutusV3 exports are available in [`packages/blaze-core/src/types.ts`](../../../packages/blaze-core/src/types.ts#L197-L312).
- The emulator validates and tracks governance proposals, voting, ratification, enactment, DRep state, committee state, deposits, expiry, and treasury effects in [`packages/blaze-emulator/src/emulator.ts`](../../../packages/blaze-emulator/src/emulator.ts).
- Governance script witnesses are accounted for with `RedeemerTag.Proposing` and `RedeemerTag.Voting` in [`packages/blaze-emulator/src/emulator.ts`](../../../packages/blaze-emulator/src/emulator.ts#L975-L1030).
- Governance RPC state and endpoints expose committee, DRep, proposal status, and tally data in [`packages/blaze-emulator/src/rpc/app.ts`](../../../packages/blaze-emulator/src/rpc/app.ts#L921-L1123).
- Emulator RPC state snapshots include governance state in [`packages/blaze-emulator/src/rpc/adapters/state.ts`](../../../packages/blaze-emulator/src/rpc/adapters/state.ts#L59-L100).
- The transaction builder includes governance transaction methods covered by the existing changesets [`hungry-flies-wink.md`](../../../.changeset/hungry-flies-wink.md) and [`weak-ghosts-care.md`](../../../.changeset/weak-ghosts-care.md).

## Technical Analysis

### Upstream Source Review

This analysis was derived from source-level review of the upstream implementations named or implied by the proposal. The exact snapshots reviewed were:

| Project | Snapshot reviewed | Relevant source inspected |
| --- | --- | --- |
| CML / `cardano-multiplatform-lib` | `a34592e1e0d3c1e958d72f05c72282d788403aa5` | Conway CDDL for governance, certificates, transaction body, multi-era Rust/WASM accessors |
| `cardano-cli` | `f017b7151459474675d0939199edafc5cc3b9dcf` | Conway governance action, vote, DRep, proposal-script, and vote-script command implementations |
| `cardano-node` / `cardano-testnet` | `a53156c39ae56b3b24bab9433def3143cea2b216` | Governance test helpers that exercise CLI flows against live epoch state |
| Mesh SDK | `3df879e48f400f19eea0ebddfb00099b0809f484` | Transaction builder governance, vote/proposal script witnesses, DRep certificates, governance types |
| Lucid Evolution | `2c9056958f8dc4e78a3207e0246f8a31bc014c2f` | Transaction builder governance, emulator docs, script-context handling, wallet witness extraction |

The common finding is that the existing libraries already cover a lot of Conway transaction construction. The part Blaze must own for this proposal is the local, deterministic governance state machine: proposal admission, vote validation, DRep and committee state, snapshots, tallies, ratification, enactment, expiry, refunds, treasury effects, and inspectable RPC state.

### Ledger And CLI Requirements

The Conway transaction format makes governance a transaction-body feature, not an off-chain convention. CML's Conway CDDL defines `voting_procedures` as voter-to-action voting maps and `proposal_procedure` as deposit, return reward account, governance action, and anchor ([governance.cddl](https://github.com/dcSpark/cardano-multiplatform-lib/blob/a34592e1e0d3c1e958d72f05c72282d788403aa5/specs/conway/governance.cddl#L2-L14)). The same CDDL enumerates the seven action families Blaze must handle: parameter change, hard fork initiation, treasury withdrawals, no confidence, committee update, new constitution, and info action ([governance.cddl](https://github.com/dcSpark/cardano-multiplatform-lib/blob/a34592e1e0d3c1e958d72f05c72282d788403aa5/specs/conway/governance.cddl#L16-L25)). Voters are not a single credential type; the CDDL distinguishes committee hot key/script, DRep key/script, and stake-pool key voters ([governance.cddl](https://github.com/dcSpark/cardano-multiplatform-lib/blob/a34592e1e0d3c1e958d72f05c72282d788403aa5/specs/conway/governance.cddl#L70-L95)).

The certificate surface expands accordingly. Conway certificates include vote delegation, stake-and-vote delegation, vote registration/delegation, committee hot authorization/resignation, DRep registration, DRep unregistration, and DRep update ([certs.cddl](https://github.com/dcSpark/cardano-multiplatform-lib/blob/a34592e1e0d3c1e958d72f05c72282d788403aa5/specs/conway/certs.cddl#L14-L28)). The DRep certificate tags carry explicit deposits and anchors where appropriate ([certs.cddl](https://github.com/dcSpark/cardano-multiplatform-lib/blob/a34592e1e0d3c1e958d72f05c72282d788403aa5/specs/conway/certs.cddl#L76-L81)), and `d_rep` values may be key, script, `always_abstain`, or `always_no_confidence` ([certs.cddl](https://github.com/dcSpark/cardano-multiplatform-lib/blob/a34592e1e0d3c1e958d72f05c72282d788403aa5/specs/conway/certs.cddl#L83-L91)).

The Conway transaction body adds `voting_procedures`, `proposal_procedures`, `current_treasury_value`, and `donation` fields ([transaction.cddl](https://github.com/dcSpark/cardano-multiplatform-lib/blob/a34592e1e0d3c1e958d72f05c72282d788403aa5/specs/conway/transaction.cddl#L27-L30)). It also adds PlutusV3 language-view handling and requires the corresponding cost model when a Plutus language is used ([transaction.cddl](https://github.com/dcSpark/cardano-multiplatform-lib/blob/a34592e1e0d3c1e958d72f05c72282d788403aa5/specs/conway/transaction.cddl#L104-L109)).

`cardano-cli` confirms how users are expected to construct these objects. Its Conway governance action runner dispatches separate commands for constitution, protocol-parameter update, treasury withdrawal, committee update, no-confidence, hard-fork, info, and view actions ([Actions/Run.hs](https://github.com/IntersectMBO/cardano-cli/blob/f017b7151459474675d0939199edafc5cc3b9dcf/cardano-cli/src/Cardano/CLI/EraBased/Governance/Actions/Run.hs#L41-L60)). The implementations create proposal procedures from deposits, return stake credentials, anchors, optional previous governance action IDs, and action-specific fields for no-confidence/new constitution ([Actions/Run.hs](https://github.com/IntersectMBO/cardano-cli/blob/f017b7151459474675d0939199edafc5cc3b9dcf/cardano-cli/src/Cardano/CLI/EraBased/Governance/Actions/Run.hs#L119-L223)), committee updates and protocol-parameter updates ([Actions/Run.hs](https://github.com/IntersectMBO/cardano-cli/blob/f017b7151459474675d0939199edafc5cc3b9dcf/cardano-cli/src/Cardano/CLI/EraBased/Governance/Actions/Run.hs#L226-L342)), and treasury withdrawal plus hard-fork initiation ([Actions/Run.hs](https://github.com/IntersectMBO/cardano-cli/blob/f017b7151459474675d0939199edafc5cc3b9dcf/cardano-cli/src/Cardano/CLI/EraBased/Governance/Actions/Run.hs#L386-L480)).

The CLI also makes governance scripts first-class purposes. Vote files can carry key witnesses, simple scripts, Plutus scripts, reference scripts, redeemers, and execution units; the code explicitly notes that script-witnessed vote files are constrained to avoid ambiguous redeemer-map construction ([Vote/Read.hs](https://github.com/IntersectMBO/cardano-cli/blob/f017b7151459474675d0939199edafc5cc3b9dcf/cardano-cli/src/Cardano/CLI/EraBased/Script/Vote/Read.hs#L27-L105)). Proposal files have equivalent simple/reference/Plutus witness support and wrap Plutus witnesses as proposing-script witnesses ([Proposal/Read.hs](https://github.com/IntersectMBO/cardano-cli/blob/f017b7151459474675d0939199edafc5cc3b9dcf/cardano-cli/src/Cardano/CLI/EraBased/Script/Proposal/Read.hs#L27-L118)). Blaze therefore has to account for `vote` and `propose` redeemer purposes, not only spend/mint/cert/withdrawal.

`cardano-testnet` shows the live-node burden this proposal is trying to remove for SDK tests. DRep registration requires CLI key generation, registration certificate creation, transaction build/sign/submit, and the current DRep deposit queried from governance state ([DRep.hs](https://github.com/IntersectMBO/cardano-node/blob/a53156c39ae56b3b24bab9433def3143cea2b216/cardano-testnet/src/Testnet/Process/Cli/DRep.hs#L63-L103), [DRep.hs](https://github.com/IntersectMBO/cardano-node/blob/a53156c39ae56b3b24bab9433def3143cea2b216/cardano-testnet/src/Testnet/Process/Cli/DRep.hs#L231-L248)). Voting requires separate vote files, a transaction body with `--vote-file`, witness overrides, signing, submission, and later lookup of the governance action ID/index in epoch state ([DRep.hs](https://github.com/IntersectMBO/cardano-node/blob/a53156c39ae56b3b24bab9433def3143cea2b216/cardano-testnet/src/Testnet/Process/Cli/DRep.hs#L147-L209), [DRep.hs](https://github.com/IntersectMBO/cardano-node/blob/a53156c39ae56b3b24bab9433def3143cea2b216/cardano-testnet/src/Testnet/Process/Cli/DRep.hs#L341-L425)). The testnet helper queries live `NewEpochState` for DRep state, governance state, DRep deposit, governance action deposit, treasury, protocol params, and governance action lifetime ([Query.hs](https://github.com/IntersectMBO/cardano-node/blob/a53156c39ae56b3b24bab9433def3143cea2b216/cardano-testnet/src/Testnet/Components/Query.hs#L515-L596), [Query.hs](https://github.com/IntersectMBO/cardano-node/blob/a53156c39ae56b3b24bab9433def3143cea2b216/cardano-testnet/src/Testnet/Components/Query.hs#L650-L672)). For Blaze M1, the emulator must give developers those state transitions in-process without standing up a node.

### CML / Serialization Library Analysis

CML provides the transaction and protocol-parameter structures Blaze should align with. Its multi-era Rust helpers expose Conway-only `voting_procedures` and `proposal_procedures` accessors ([utils.rs](https://github.com/dcSpark/cardano-multiplatform-lib/blob/a34592e1e0d3c1e958d72f05c72282d788403aa5/multi-era/rust/src/utils.rs#L746-L766)). It exposes governance action validity period, governance action deposit, DRep deposit, and DRep inactivity period as Conway protocol-parameter update fields ([utils.rs](https://github.com/dcSpark/cardano-multiplatform-lib/blob/a34592e1e0d3c1e958d72f05c72282d788403aa5/multi-era/rust/src/utils.rs#L1179-L1212)). The WASM helpers expose Conway certificate downcasts for vote delegation and DRep certificates ([utils.rs](https://github.com/dcSpark/cardano-multiplatform-lib/blob/a34592e1e0d3c1e958d72f05c72282d788403aa5/multi-era/wasm/src/utils.rs#L307-L383)), plus transaction accessors for voting/proposal procedures ([utils.rs](https://github.com/dcSpark/cardano-multiplatform-lib/blob/a34592e1e0d3c1e958d72f05c72282d788403aa5/multi-era/wasm/src/utils.rs#L649-L654)).

What CML does not provide is a local ledger-state emulator. It can tell Blaze what a Conway transaction contains and how to serialize/inspect those fields; it does not maintain DRep registration state, committee hot credentials, stake snapshots, proposal expiry, ratification thresholds, enactment queues, deposit refunds, or treasury withdrawals. Blaze should therefore use CML/Cardano SDK structures at the boundary and implement the deterministic governance state machine above them.

### Mesh Analysis

Mesh has substantial builder support that sets the ergonomic bar for Blaze. Its V2 builder has explicit PlutusV3 spend/mint/withdraw methods and `votePlutusV1`, `votePlutusV2`, and `votePlutusV3` helpers ([mesh-tx-builder-v2/index.ts](https://github.com/MeshJS/mesh/blob/3df879e48f400f19eea0ebddfb00099b0809f484/packages/mesh-transaction/src/mesh-tx-builder-v2/index.ts#L239-L335)). The core builder supports voting scripts, reference scripts, vote redeemers, proposal creation, proposal scripts, proposal reference scripts, and proposal redeemers ([tx-builder-core.ts](https://github.com/MeshJS/mesh/blob/3df879e48f400f19eea0ebddfb00099b0809f484/packages/mesh-transaction/src/mesh-tx-builder/tx-builder-core.ts#L941-L1235)). It also has DRep registration, deregistration, and update certificate builder methods ([tx-builder-core.ts](https://github.com/MeshJS/mesh/blob/3df879e48f400f19eea0ebddfb00099b0809f484/packages/mesh-transaction/src/mesh-tx-builder/tx-builder-core.ts#L1322-L1378)).

Mesh also implements transaction-construction details Blaze should not ignore: it sorts voters by committee/DRep/SPO order and then by credential details while leaving proposals in insertion order ([mesh-tx-builder/index.ts](https://github.com/MeshJS/mesh/blob/3df879e48f400f19eea0ebddfb00099b0809f484/packages/mesh-transaction/src/mesh-tx-builder/index.ts#L447-L523)), produces `VOTE` redeemer tags with indexes for script votes ([mesh-tx-builder/index.ts](https://github.com/MeshJS/mesh/blob/3df879e48f400f19eea0ebddfb00099b0809f484/packages/mesh-transaction/src/mesh-tx-builder/index.ts#L592-L615)), derives required witnesses for DRep certificates and basic votes ([mesh-tx-builder/index.ts](https://github.com/MeshJS/mesh/blob/3df879e48f400f19eea0ebddfb00099b0809f484/packages/mesh-transaction/src/mesh-tx-builder/index.ts#L1158-L1176), [mesh-tx-builder/index.ts](https://github.com/MeshJS/mesh/blob/3df879e48f400f19eea0ebddfb00099b0809f484/packages/mesh-transaction/src/mesh-tx-builder/index.ts#L1238-L1262)), and adds DRep/proposal deposits into transaction deposit accounting ([mesh-tx-builder/index.ts](https://github.com/MeshJS/mesh/blob/3df879e48f400f19eea0ebddfb00099b0809f484/packages/mesh-transaction/src/mesh-tx-builder/index.ts#L1294-L1312)). Its type layer also models governance proposal status fields, protocol-parameter governance thresholds, deposits, action validity period, and DRep inactivity ([governance.ts](https://github.com/MeshJS/mesh/blob/3df879e48f400f19eea0ebddfb00099b0809f484/packages/mesh-common/src/types/governance.ts#L5-L150)).

Mesh remains a transaction builder and chain-client toolkit. It does not replace a deterministic emulator that can ratify/enact proposals locally. Blaze must match the important builder ergonomics and witness/deposit behavior, then go deeper by applying the resulting governance transactions to emulator state.

### Lucid Evolution Analysis

The implementation records DRep registration and deregistration actions ([TxBuilder.ts](https://github.com/Anastasia-Labs/lucid-evolution/blob/2c9056958f8dc4e78a3207e0246f8a31bc014c2f/packages/lucid/src/tx-builder/TxBuilder.ts#L937-L1008)), treasury donation ([TxBuilder.ts](https://github.com/Anastasia-Labs/lucid-evolution/blob/2c9056958f8dc4e78a3207e0246f8a31bc014c2f/packages/lucid/src/tx-builder/TxBuilder.ts#L1099-L1124)), vote delegation and register-and-delegate flows ([TxBuilder.ts](https://github.com/Anastasia-Labs/lucid-evolution/blob/2c9056958f8dc4e78a3207e0246f8a31bc014c2f/packages/lucid/src/tx-builder/TxBuilder.ts#L1186-L1303)). Lucid also translates Conway DRep/delegation certificates into Plutus script context data ([ScriptContext.ts](https://github.com/Anastasia-Labs/lucid-evolution/blob/2c9056958f8dc4e78a3207e0246f8a31bc014c2f/packages/lucid/src/script-context/ScriptContext.ts#L266-L420)) and extracts required witness key hashes from vote delegation and DRep certificates ([wallet.ts](https://github.com/Anastasia-Labs/lucid-evolution/blob/2c9056958f8dc4e78a3207e0246f8a31bc014c2f/packages/wallet/src/wallet.ts#L188-L285)). The important limitation is explicit in Lucid's emulator docs: the emulator is simplified and does not implement all Cardano features, naming complex governance operations as an example ([emulator.mdx](https://github.com/Anastasia-Labs/lucid-evolution/blob/2c9056958f8dc4e78a3207e0246f8a31bc014c2f/docs/pages/documentation/deep-dives/emulator.mdx#L177-L191)). So Lucid is a strong builder/API reference, not evidence that the Blaze emulator problem is already solved elsewhere.

### Design Decisions From The Review

1. Keep governance on the normal transaction-builder path. Mesh and Lucid both expose governance as ordinary transaction composition, so Blaze should not hide M1 behavior behind test-only helpers.
2. Validate governance witnesses before mutating emulator state. CLI and Mesh distinguish vote/propose script witnesses and redeemers; Blaze should reject missing or misplaced redeemers at the same boundary as other script witnesses.
3. Use explicit epoch advancement. `cardano-testnet` must wait/query live epoch state; Blaze should make that deterministic with `stepForwardToNextEpoch()` so tests can assert exact snapshot, ratification, enactment, expiry, and refund behavior.
4. Treat CML/Cardano SDK types as the serialization boundary, not as the emulator. CML gives the Conway shapes; Blaze must supply local ledger-state effects.
5. Expose RPC state for external harnesses. A proposal reviewer should be able to inspect DReps, committee, proposal status, tallies, treasury, and snapshots without opening private emulator internals.

### Planned/Accepted M1 Feature Set

For M1, the accepted feature set is:

- Governance action submission for parameter changes, treasury withdrawals, committee updates, new constitution, no confidence, hard fork initiation, and info actions.
- DRep registration, re-registration, deregistration/expiry handling, vote delegation, key DRep voting, and script DRep voting.
- Committee membership, quorum, and hot credential handling.
- Stake snapshot construction, DRep/SPO/committee tallying, threshold checks, ratification, enactment, delayed enactment for no-confidence, expiry, and deposit refunds.
- Proposal-policy script witness validation for parameter-change and treasury-withdrawal actions.
- PlutusV3-capable script handling through core exports and transaction witness serialization.
- RPC endpoints for governance state inspection and test orchestration.
- Benchmarks and executable docs that prove the model is usable as an SDK test harness.

Out of M1 scope:

- Bit-for-bit reimplementation of every `cardano-node` ledger rule.
- Network synchronization, mempool behavior, or chain selection.
- Live-node governance submission automation.
- Full formal proof against the Conway ledger spec.

This scope is deliberately narrower than `cardano-node`, but deeper than a transaction-builder comparison: it targets the stateful governance behavior that dApp and SDK developers need to test locally.

## Milestone scope

## Benchmark comparison

The M4 benchmark repeats the command and emulator scenarios from the M1 baseline.

| Scenario | M1 latency avg | M4 latency avg | Change | M1 throughput avg | M4 throughput avg | Change |
| --- | ---: | ---: | ---: | ---: | ---: | ---: |
| Register mocked wallet | 269,530 ns | 135,223 ns | 49.8% faster | 3,829 ops/s | 7,629 ops/s | 99.2% higher |
| Simple payment | 2,757,496 ns | 2,317,214 ns | 16.0% faster | 418 ops/s | 497 ops/s | 18.9% higher |
| Script lock -> spend | 11,939,709 ns | 9,638,712 ns | 19.3% faster | 92 ops/s | 113 ops/s | 22.8% higher |

The parameter change governance flow, added after M1, averaged 8,634,841 ns and 135 ops/s in the same run.

## Acceptance criteria

- The emulator can run as a standalone HTTP server and can be embedded through `startRpcServer`.
- Clients can reset the emulator with `mainnet`, `preprod`, `preview`, or custom network settings.
- Clients can configure slot duration, slots per block, slots per epoch, and protocol parameters, then inspect the active settings through `/emulator/config` and `/emulator/parameters`.
- Transactions can be submitted as raw CBOR through `/emulator/transactions`. The RPC tests cover a signed transaction built with lower-level Cardano serialization primitives instead of Blaze `TxBuilder`, and callers can use the same endpoint with Haskell tooling, another TypeScript builder, or a CLI harness.
- The RPC server exposes OpenAPI metadata and Swagger UI.
- The docs explain server startup, health checks, wallet setup, raw CBOR submission, network configuration, script publishing, governance utilities, and cleanup.
- The M1 registration, payment, and script round-trip benchmark scenarios all improve in average latency and throughput.
